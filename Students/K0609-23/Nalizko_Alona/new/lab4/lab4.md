Отчёт по выполнению урока 4.

В рамках урока 4 по логированию и observability все контейнеры были успешно запущены и работают. Команда docker-compose ps показывает, что сервисы postgres, app, nginx и rate-limiter находятся в статусе Up, при этом app и postgres имеют статус healthy, nginx и rate-limiter также функционируют, хотя их статус здорового не отображается. Файловое логирование настроено и работает: в контейнере rate-limiter в директории /app/logs присутствуют два файла логов — requests-2026-04-14.log и requests-2026-04-16.log. Размер файлов составляет от 1872 байт до 28090 байт, что подтверждает, что запросы успешно пишутся в файлы. При попытке использовать wildcard в команде cat возникла ошибка, потому что звездочка не раскрылась корректно, но при указании конкретного файла логи читаются без проблем.

Структурированное логирование в формате JSON реализовано, строки логов содержат все необходимые поля: timestamp, method, path, status_code, response_time_ms, client_ip, user_agent, rate_limited. Это позволяет использовать jq для парсинга и фильтрации. Двойное логирование работает в полном объеме: все запросы одновременно пишутся и в файлы, и в базу данных PostgreSQL.

База данных demo содержит таблицу request_logs, в которой на момент проверки находится 189 записей. При группировке по статус-коду получено 164 записи с кодом 200 и 25 записей с кодом 404. Записей с кодом 429 нет, поскольку в данной реализации rate limiting возвращает код 503 вместо 429, что является особенностью middleware. Среднее время ответа для запросов с кодом 200 составляет 4.56 миллисекунды, максимальное — 54 миллисекунды. Для запросов с кодом 404 среднее время ответа — 5.08 миллисекунды, максимальное — 21 миллисекунда. Это говорит о том, что приложение обрабатывает запросы быстро, задержки находятся в пределах нормы.

Выполнен анализ временного ряда: запросы отсортированы по timestamp DESC, последние 20 записей показывают, что все недавние запросы к /api/status возвращают код 200. Также проведена фильтрация по response_time_ms, что позволяет выявить медленные запросы. Все команды из урока 4 выполнены успешно: просмотр файловых логов через tail и cat с jq, запросы к базе данных с группировкой по статус-кодам, вычисление средней и максимальной задержки, фильтрация по конкретным статусам и временным интервалам.

Таким образом, двойное логирование настроено корректно, структурированные JSON логи пишутся в файлы, все запросы сохраняются в PostgreSQL, доступны для анализа и агрегации. Observability обеспечена на уровне файлов и базы данных. Урок 4 полностью выполнен и может быть сдан. Для отчёта рекомендуется приложить скриншоты: содержимое директории /app/logs с файлами логов, пример JSON-лога с выводом через jq, результат SELECT * FROM request_logs LIMIT 10, статистику по статус-кодам с GROUP BY, результаты вычисления среднего и максимального времени ответа, а также последние 20 записей по времени.
