## Отчет отладка и Network Debugging

Для автоматизации подготовки рабочей среды используется скрипт `setup-debugging.sh`. Он выполняет последовательную установку инструментов отладки (`tcpdump`, `strace`, `curl`, `iproute2`) во все активные контейнеры проекта (`practic-app-1`, `practic-nginx-1`, `practic-postgres-1`) через команды `docker exec` и менеджер пакетов `apk`. Это обеспечивает наличие необходимого инструментария для проведения сетевого анализа и имитации сбоев без необходимости изменения исходных Docker-образов.

### Подготовка окружения и инструментов отладки
В ходе настройки `docker-compose.yml` выявляются и исправляются ошибки форматирования YAML, связанные с некорректными отступами и вложенностью сервисов. Для обеспечения отказоустойчивости в конфигурацию базы данных `postgres` добавляется блок `healthcheck` с использованием утилиты `pg_isready`, а для зависимых сервисов прописывается условие `condition: service_healthy`. Это гарантирует, что приложение и прокси-слои начинают работу только после полной готовности СУБД к приему соединений.

### Сценарий 1 и 3: Сетевой анализ и трассировка системных вызовов
С помощью `tcpdump` выполняется перехват и анализ трафика на различных интерфейсах. На порту 53 фиксируются DNS-запросы и ответы (записи типа A и AAAA) для внешних доменов. На порту 5432 анализируется протокол PostgreSQL, где отслеживаются SQL-запросы (например, `SELECT COUNT(*) FROM users`) и ответы сервера. На порту 3000 фиксируется полный цикл TCP-соединения между Nginx и Middleware: установка (`Flags [S]`), передача полезной нагрузки (`Flags [P.]`) и завершение сессии (`Flags [F.]`).

Для низкоуровневого анализа сетевого стека используется `strace`. Трассировка системных вызовов команды `curl` позволяет отследить этапы создания сокета, вызов `connect` (возвращающий `EINPROGRESS` для неблокирующих операций) и обмен данными через `sendto` и `recvfrom`. Зафиксированы как успешные ответы (`200 OK`), так и ошибки отсутствия ресурса (`404 Not Found`).

### Сценарий 2: Имитация отказа базы данных
Выполняется сценарий преднамеренного повреждения структуры данных. Скрипт `break-db.sh` производит удаление таблицы `users` командой `DROP TABLE IF EXISTS users CASCADE`. Это приводит к возникновению ошибки `42P01` (relation "users" does not exist) в логах приложения `practic-app-1`. Работоспособность восстанавливается путем ручного воссоздания структуры таблицы через SQL-команду `CREATE TABLE` внутри контейнера базы данных. Последующая проверка через `curl` подтверждает корректный возврат данных в формате JSON.

### Сценарий 4: Эмуляция высокой задержки
Реализуется сценарий деградации производительности сети с помощью утилиты `tc` (Traffic Control). При первичной попытке внедрения задержки фиксируется ошибка `RTNETLINK answers: Operation not permitted`. Проблема решается путем добавления привилегии `cap_add: NET_ADMIN` в конфигурацию контейнера в `docker-compose.yml`. 

На интерфейс `eth0` контейнера базы данных накладывается задержка в **100мс** через дисциплину очереди `netem`. Наличие задержки подтверждается утилитой `ping`, показывающей среднее время отклика **100.8 мс**. Общее время выполнения HTTP-запроса, измеренное через `curl`, составляет **0.115с**, что свидетельствует об успешном мониторинге сетевых задержек на уровне Middleware. Для завершения эксперимента правила `tc` удаляются, и фиксируется возврат системы к штатным показателям времени отклика (около **0.01с**).

### Решенные в ходе работы технические ошибки
* **Ошибки конфигурации Docker:** Исправлена структура YAML-файла, настроены зависимости `depends_on` с учетом состояния здоровья контейнеров.
* **Ошибки прав доступа:** Реализована возможность управления сетевыми параметрами ядра через добавление Linux-capabilities (`NET_ADMIN`).
* **Ошибки отсутствия пакетов:** В минималистичные Alpine-образы внедрены необходимые бинарные файлы для диагностики сети.
* **Сетевые коллизии:** Решена ошибка `Exclusivity flag on` при настройке `tc` путем использования команды `replace` вместо `add`.
