# Модуль 1 — Процессы, Демоны & systemd

## Задача
Написать systemd unit для произвольного скрипта, который должен автоматически перезапускаться при падении.

## Решение
- Написан скрипт-воркер на Python, выводящий свой текущий PID и пишущий метрики в стандартный вывод.
- С помощью редактора `vim` создан конфигурационный `.service` файл в `/etc/systemd/system/`.
- В unit-файле настроена директива `Restart=always` для обеспечения отказоустойчивости.
- Обновлена конфигурация демонов (`systemctl daemon-reload`), сервис добавлен в автозагрузку и запущен.
- Для проверки поведения системы процесс был найден через `ps` и принудительно завершен сигналом `SIGKILL`.
- Корректный рестарт процесса под новым PID зафиксирован через `systemctl status` и чтение логов `journalctl`.

## Команды которые открыл для себя
- `ps -ef --forest` — выводит процессы в виде дерева. Помогает наглядно увидеть иерархию (кто кого породил, связь PID и PPID).
- `:%s/old/new/g` (в нормальном режиме Vim) — глобальная автозамена текста по всему файлу. Удобно при быстром редактировании конфигов.
- `kill -9 <PID>` — отправка сигнала SIGKILL напрямую ядру. Безусловное завершение процесса, который невозможно перехватить.

## Сложности
- **Ошибка E212 в Vim:** забыл использовать `sudo` при создании файла скрипта в системной директории `/opt/`. Пришлось выходить без сохранения и переоткрывать редактор с нужными правами.
- **CrashLoop сервиса:** из-за того, что файл скрипта изначально не сохранился, `systemd` пытался запускать несуществующий файл, мгновенно падал и уходил в бесконечный цикл рестартов. Проблема была выявлена через `journalctl` и решена корректным созданием файла.
- **Фантомный процесс:** при поиске PID скрипта командой `ps aux | grep metrics-worker` я увидел строку процесса и попытался убить его, но получил ошибку "No such process". Оказалось, я пытался убить саму отработавшую команду `grep`.


# Практика 01 — Топ-10 IP из Nginx access.log

## Задача
Вывести топ-10 IP-адресов по числу запросов из сгенерированного файла `access.log` (1 млн строк), а также из его JSON-версии.

## Решение
- Исправлена синтаксическая ошибка в предоставленном bash-скрипте генератора логов. Скрипт успешно отработал и создал файлы `access.log` и `access_json.log`.
- Для классического лога применен конвейер утилит: парсинг первого столбца, алфавитная сортировка, подсчет дубликатов, числовая сортировка и обрезка до нужного количества строк.
- Для JSON-версии лога вместо обработки текста по столбцам использован парсер JSON для извлечения значений по конкретному ключу, с последующей передачей в тот же сортировочный конвейер.
- Сгенерированные данные очищены с помощью `rm -f`.

## Команды которые открыл для себя
- `awk '{print $1}'` — читает вывод построчно и выводит только первый столбец текста (разделитель — пробел). Базовый инструмент для парсинга классических логов веб-серверов.
- `uniq -c` — схлопывает идущие подряд одинаковые строки и считает количество их вхождений (поэтому применяется только после `sort`).
- `jq -r '.key_name'` — легковесная утилита для работы с JSON в терминале. Вытаскивает сырое значение по нужному ключу (флаг `-r` убирает лишние кавычки из вывода).

## Сложности
- **Ошибка в исходном скрипте:** генератор логов Nginx падал на этапе создания классического лога с ошибкой `NameError: name 'i' is not defined`. Проблема оказалась внутри Python-кода (list comprehension), где в генерации пула IP адресов в цикле использовался `_`, но при этом вызывалась переменная `i`. Исправил `for _ in range(n)` на `for i in range(n)`.
