# Модуль 4: Сеть — curl, ss, tcpdump и перехват трафика
## Задача
Использовать сниффер tcpdump для захвата HTTP-трафика к google.com. Для этого нужно было разделить экран терминала через tmux, в одном окне запустить перехват пакетов, а в соседнем — сделать веб-запрос с помощью curl -v. Главная цель: найти в дампе "тройное рукопожатие" (TCP handshake: SYN, SYN-ACK, ACK) и прочитать HTTP-заголовки ответа серверами.

## Решение
Я создала две панели в tmux. В левой запустила прослушивание 80-го порта (HTTP), а в правой сымитировала клиентский запрос. Сразу после этого в левой панели отобразился весь процесс установки соединения и передачи данных.

Команды:

<pre>

1. В левой панели запускаем ловушку для трафика:
-i any (слушать все сетевые интерфейсы)
-n (не тратить время на перевод IP-адресов в доменные имена)
-A (выводить содержимое пакетов в текстовом виде ASCII)
sudo tcpdump -i any port 80 and host google.com -n -A

2. В правой панели делаем подробный веб-запрос:
curl -v http://google.com

3. После перехвата останавливаем tcpdump комбинацией Ctrl+C.
</pre>

## Команды и понятия, которые открыла для себя
TCP Handshake — то самое "тройное рукопожатие", база любого сетевого соединения. В дампе я четко увидела все три этапа:

Flags [S] (SYN) — мой компьютер просит Гугл открыть соединение.

Flags [S.] (SYN-ACK) — Гугл отвечает, что готов.

Flags [.] (ACK) — мой комп подтверждает прием, соединение установлено.

tcpdump — невероятно мощная утилита, которая позволяет заглянуть "под капот" сети и увидеть каждый пролетающий байт.

curl -v — ключ -v (verbose) заставляет утилиту выводить не только скачанный HTML-код, но и весь процесс общения с сервером (отправленные и полученные заголовки).

## Сложности
Главной неочевидной хитростью было то, что запрос нужно делать именно на http://google.com, а не на защищенный https://. Сначала я по привычке пыталась поймать трафик на 443 порту по защищенному протоколу. В итоге tcpdump честно перехватывал пакеты, но из-за TLS-шифрования флаг -A выводил на экран просто нечитаемую абракадабру, и найти там текстовые HTTP-заголовки было невозможно.

Как только я переключила фильтр на port 80 и сделала запрос по открытому HTTP — в консоли сразу появился чистый и понятный текст моего запроса GET / HTTP/1.1 и ответа сервера HTTP/1.1 301 Moved Permanently. А еще я увидела, как в конце машины обменялись пакетами с флагом [F.] (FIN), вежливо разорвав соединение.


# ЧАСТЬ 4.2: Break Lab — Диагностика незапускаемых бинарников
## Задача
Учебный скрипт создал в папке /opt/break_lab несколько файлов-обманок. Все они отказывались запускаться, выдавая совершенно разные ошибки системы. Мне нужно было использовать диагностические утилиты Linux, чтобы понять причину поломки каждого файла и, по возможности, заставить их работать.

## Решение
Я по очереди пыталась запустить каждый файл, анализировала вывод ошибок и использовала команду file, чтобы заглянуть "внутрь" файлов без их выполнения.

Команды:

<pre>

1. Запустила скрипт и перешла в папку с проблемами:
sudo bash 04_binary_break.sh
cd /opt/break_lab

2. Починила файл без прав (выдавал Permission denied):
file mystery_no_exec           # Убедилась, что это программа
sudo chmod +x mystery_no_exec  # Добавила права на выполнение
./mystery_no_exec              # Успешный запуск!

3. Починила скрипт с кривым интерпретатором (bad interpreter):
cat mystery_bad_interpreter    # Посмотрела исходный код
sudo nano mystery_bad_interpreter

(Исправила первую строку shebang на #!/bin/bash)
./mystery_bad_interpreter      # Успешный запуск!

4. Диагностировала физически сломанный файл (Exec format error):
file mystery_truncated_elf

(Утилита показала, что у файла отсутствуют важные заголовки, файл обрезан и восстановлению не подлежит).
</pre>

## Команды и понятия, которые открыла для себя
file — самая главная утилита для диагностики. В Линуксе расширения файлов (типа .exe или .txt) вообще ничего не значат. Файл может называться как угодно, но команда file читает его "заголовки" (магические числа) и точно говорит, что это: текст, картинка или скомпилированная программа (ELF).

Shebang (#!) — самая первая строчка в любом текстовом скрипте (например, #!/bin/bash). Она говорит Линуксу, какую программу использовать, чтобы прочитать этот файл. Если там написана чушь или путь к несуществующей программе, скрипт вывалит ошибку bad interpreter.

chmod +x — добавление бита исполнения. В Линуксе нельзя просто так взять и запустить файл, даже если это стопроцентная программа. Пока не дашь права на запуск (execute), система будет бить по рукам.

ldd и strace — мощные инструменты для более сложного дебага. ldd показывает, каких динамических библиотек (.so файлов) не хватает программе для старта, а strace отслеживает все системные вызовы программы в реальном времени.

## Сложности
Интересный опыт — понять, что не каждую ошибку можно "починить". Когда я пыталась запустить mystery_truncated_elf, система выдала Exec format error. Сначала я думала, что нужно поменять права или владельца, но утилита file показала, что бинарник просто обрезан (не хватает секций). Это отличный урок: если бинарный файл скачался с обрывом интернета или повредился на диске, никакие настройки прав его не спасут, нужна переустановка.
