# Модуль 2 — Диски, Файловая Система и магия inode
## Задача
Нужно было зайти в папку /data и навести там суету: найти все файлы логов (.log), которые лежат там дольше 7 дней и весят больше 100 МБ. А потом одним махом удалить их через утилиту xargs и посчитать, сколько места на диске я освободила. Заодно нужно было пощупать команды для диагностики дисков и файлов.

## Решение
Сначала я проверила, сколько места жрет папка до чистки, почекала метаданные файлов, а потом запустила поиск с удалением.

Команды:

<pre>

1. Посмотрела, сколько весит папка изначально, и отсортировала файлы по размеру:
du -sh /data
du -sh /data/* | sort -rh

2. Глянула статистику по айнодам (inode) и досье на конкретный файл:
df -i /data
stat /data/old_big1.log

3. Нашла старые и жирные логи, а потом сразу передала их на удаление:
find /data -type f -name "*.log" -mtime +7 -size +100M | xargs rm -v

4. Проверила, сколько места освободилось после чистки:
du -sh /data
</pre>

## Команды, которые открыла для себя
du -sh — показывает, сколько реально весит папка (ключ s собирает всё в сумму, а h выводит в нормальном человеческом виде — мегабайтах и гигабайтах, а не в байтах).

df -i — показывает свободное место не в гигабайтах, а в свободных айнодах (inode). inode — это такая карточка файла в системе, где лежат его права, вес и дата создания. Бывает так, что гигабайты на диске есть, а айноды кончились (например, создали миллион пустых файлов), и система падает.

stat — выдает полное досье на файл: когда он был создан, когда его последний раз открывали, какой у него номер inode.

find с кучей ключей — мощнейший поисковик. -mtime +7 ищет то, что меняли больше 7 дней назад, а -size +100M ищет жирные файлы больше ста метров.

xargs — работает как крутая труба. Он берет список файлов, который нашел find, и по очереди подставляет их в команду rm, чтобы грохнуть всё за один присест.

## Сложности
Само удаление прошло как по маслу, а вот на этапе подготовки тестовых файлов я попалась в классическую ловушку Линукса с правами доступа!

Я создала директорию /data через sudo mkdir, и её владельцем автоматически стал root. А потом попыталась создать там фейковый старый файл хитрой склейкой: sudo fallocate -l 150M /data/old_big1.log && touch -d "10 days ago" /data/old_big1.log.

В итоге первая часть команды (fallocate) отработала с правами админа и создала файл, а вот вторая часть (touch — чтобы подделать дату) запустилась уже от моего обычного пользователя. Линукс тут же ударил меня по рукам: «невозможно выполнить touch... Отказано в доступе»!

Чтобы починить этот косяк, мне пришлось забрать права на папку себе командой sudo chown -R $USER:$USER /data. После этого Линукс перестал ругаться, файлы создались, и я спокойно снесла их через find + xargs.

# ЧАСТЬ 2.2: Break Lab — Убиваем и чиним сеть
## Задача
Нужно было запустить скрипт, который ломает сеть на виртуалке (удаляет маршруты и ломает резолв доменов). После этого интернет полностью отваливается, и сайты не пингуются. Моя задача была — руками найти, где скрипт нагадил, и починить конфиги, чтобы curl https://www.avito.ru снова начал работать.

## Решение
Скрипт сломал три вещи: удалил default route, прописал левый DNS-сервер и запретил системе использовать DNS для поиска адресов. Я починила всё по очереди.

Команды:

<pre>

1. Запустила скрипт-вредитель и убедилась, что Авито не открывается
sudo bash 02_network_break.sh
curl -I --max-time 5 https://www.avito.ru

2. Починила файл, который говорит системе, ГДЕ искать IP-адреса сайтов
sudo nano /etc/nsswitch.conf

(исправила строку 'hosts: files' на 'hosts: files dns')
3. Починила файл, где указан сам DNS-сервер
sudo nano /etc/resolv.conf

(удалила сломанный IP 203... и прописала 'nameserver 8.8.8.8')
4. Восстановила удаленный маршрут (default route) из бэкапа, который оставил скрипт
sudo ip route restore < /var/lib/linux_break_lab/routes.tab

5. Снова проверила Авито — интернет появился!
curl -I https://www.avito.ru
</pre>

## Команды и файлы, которые открыла для себя
/etc/nsswitch.conf — база. Этот файл говорит Линуксу, в каком порядке искать информацию. Если в строке hosts стоит только files, система будет смотреть только в локальный файлик /etc/hosts и даже не попытается пойти в интернет. Я добавила туда dns, чтобы она начала спрашивать адреса у DNS-сервера.

/etc/resolv.conf — файл, где лежит адрес самого DNS-сервера. Скрипт подсунул туда фейковый IP, а я вернула нормальный гугловский (8.8.8.8).

ip route del default — так скрипт убил мне выход в сеть. Компьютер просто перестал понимать, в какую дверь (шлюз) отправлять пакеты.

ip route restore < файл — крутая штука, чтобы моментально загрузить обратно все маршруты из сохраненного файла-бэкапа.

## Сложности
Это было реально стрессово! Главная сложность и страх — это то, что я работаю по SSH. Когда скрипт удаляет default route, ты можешь просто потерять соединение с сервером (консоль зависнет и выкинет тебя), потому что сервер забудет маршрут до твоего домашнего компа. Мне повезло, что обошлось.

Ну и сам ремонт заставил попотеть: нельзя просто вернуть маршрут, нужно было починить именно цепочку. Без исправленного nsswitch.conf система бы даже не прочитала resolv.conf. Пришлось чинить всё последовательно.
