# Отчет по практической работе: Логирование и Observability (SRE)

## Цель работы
Реализовать систему двойного структурированного логирования (файловая система + реляционная база данных) для обеспечения наблюдаемости сервиса ограничения частоты запросов (Rate Limiter).
---

## Выполненные шаги

1. Настройка среды: Подготовлена инфраструктура Docker Compose, включающая сервисы Nginx, Rate-Limiter (Node.js), App (Go) и PostgreSQL.
2. Реализация структурированного логирования: Настроен экспорт логов в формате JSON. Каждый лог содержит метку времени, метод, путь, статус-код, время ответа (ms), IP клиента и флаг срабатывания лимит.
3. Тестирование системы: Проведено нагрузочное тестирование с помощью скрипта `test-rate-limiting.sh`, подтвердившее корректную работу алгоритма Token Bucket (переход ответов со статуса 200 на 429 при превышении лимита).

## Cтатистика (Observability)
По результатам финального анализа через SQL-запрос подтверждена целостность данных:
- Статус 200: подтверждает доступность приложения.
- Статус 429: подтверждает работу лимитера.
- Среднее время ответа: зафиксировано в пределах нормы (1–3 ms).

**Вывод:** Система приведена в соответствие с требованиями SRE-практик, обеспечивая прозрачность работы сервиса на всех уровнях.

## Ошибки
1. Проблема раскрытия маски (Globbing) в оболочке Zsh

* Причина: Локальный терминал (zsh) пытался найти файлы со звездочкой * в текущей папке на компьютере до того, как команда была передана внутрь Docker-контейнера.
* Решение: Использование кавычек '...' или флага sh -c, чтобы маска пути обрабатывалась внутри контейнера, а не локальной оболочкой.

2. Ошибка отсутствия TTY при перенаправлении ввода

* Причина: Команда docker compose exec по умолчанию ожидает интерактивный режим (псевдо-терминал), который несовместим с передачей SQL-запросов через конструкцию << EOF (heredoc).
* Решение: Добавление флага -T для отключения выделения TTY, что необходимо для автоматизации скриптов и передачи потоков данных.

3. Ошибка авторизации в базе данных (Неверная роль)

* Причина: Попытка подключения к PostgreSQL под пользователем demo, в то время как в конфигурации docker-compose.yml суперпользователем был назначен postgres.
* Решение: Смена флага пользователя на -U postgres.
