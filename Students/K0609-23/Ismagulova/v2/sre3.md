# Отчёт по лабораторной работе: Rate Limiting (Token Bucket)

## Цель работы

Тестирование механизма ограничения частоты запросов, реализованного по алгоритму **Token Bucket** с параметрами: 10 токенов, пополнение 5 токенов в секунду.
---

## Ход выполнения

#### Выполненные команды

Тестирование выполнялось с помощью скрипта `rate-limiting/test-rate-limiting.sh`:

```bash
bash rate-limiting/test-rate-limiting.sh
```

#### Основные этапы тестирования:
1. Стресс-тест (Rapid Requests): Последовательно отправляется заданное количества GET-запросов (по умолчанию 50) с минимальной задержкой.
2. Анализ результатов: Автоматически считаются распределения статус-кодов (200 OK vs 429 Too Many Requests) и вычисление процента успешных запросов (Success Rate).
3. Тест на восстановление (Recovery Test): Три секунды ожидания для регенерации токенов в корзине, после чего отправляется проверочная серия запросов. Подтверждает, что система корректно снимает блокировку по истечении времени.
4. Конкурентный тест (Concurrent Requests): Параллельно запускаются несколько потоков для проверки поведения системы при одновременном обращении нескольких «клиентов».
5. Проверка изоляции по IP: Имитируются запросы с разных IP-адресов (через заголовок X-Real-IP).

## Ошибки
1. Ошибки парсинга вывода curl, некорректно извлечение HTTP-кода (например, получение пустого значения или 000).
    Причина: Если сервер присылал дополнительные заголовки или пустые строки, нумерация строк сбивалась.
    Решение: Использовались более надежные методы извлечения через sed с четко заданными разделителями, а также добавление флага -4 для принудительного использования протокола IPv4 (чтобы избежать задержек при резолве IPv6).
2. Проблема последовательного выполнения запросов, лимитер мог не срабатывать, так как запросы шли слишком медленно.
    Причина: Обычный цикл for в Bash ждет завершения предыдущего curl, прежде чем начать следующий.
    Решение: Добавляется блок Concurrent Requests (параллельные запросы) с использованием символа &. Это позволило отправлять несколько запросов одновременно, создавая реальную нагрузку для проверки «всплеска».
