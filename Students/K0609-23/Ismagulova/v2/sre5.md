## 1. Сетевая отладка и анализ трафика (tcpdump)

В ходе работы был проведен захват пакетов на различных уровнях взаимодействия сервисов:

*   **Взаимодействие Nginx ↔ Middleware:** С помощью команды `tcpdump -i eth0 -A 'tcp port 3000'` выполняется анализ полного цикла HTTP-запроса.
*   **Мониторинг БД:** Выполняется захват трафика на стороне PostgreSQL (порт 5432), что позволило подтвердить наличие сетевой связности между приложением и базой данных.
*   **DNS-анализ:** Проверяются UDP-запросы на порт 53 для подтверждения корректной работы внутреннего DNS Docker-сети при разрешении имен сервисов (например, `postgres`, `app`).

## 2. Тестирование API и отладка через curl

Изучены и применены расширенные возможности `curl` для диагностики:
*   **Диагностика заголовков:** Использование флагов `-v` (verbose) и `-i` (include headers) для анализа метаданных ответов и проверки работы Rate Limiter через кастомные заголовки `X-RateLimit-*`.
*   **Профилирование:** Применение формата `-w` для замера общего времени ответа (`time_total`), что необходимо для поиска узких мест в производительности.
*   **Автоматизация тестов:** Написаны циклы для имитации нагрузки, проверяющие восстановление токенов (Token Refill) в алгоритме ограничения частоты запросов.
*   **Работа с состоянием и сессиями:** Использование флага `-b` для передачи куки (`session=abc123`), что позволило проверить корректность обработки сессионных данных в прокси-слое.
*   **Тестирование POST-запросов:** Отправка JSON-данных через `-X POST -d`, имитирующая взаимодействие фронтенда с API.
*   **Обеспечение отказоустойчивости (Resilience):** 
    *   Применение флагов `--retry` и `--retry-delay` для проверки поведения клиента при кратковременных сбоях сети.
    *   Использование `--max-time` для тестирования тайм-аутов, чтобы предотвратить зависание клиентских запросов при «зависании» бэкенда.
*   **Форматирование вывода:** Настройка кастомного формата ответа (`-w`) для одновременного мониторинга HTTP-статуса и времени ответа (`time_total`) в структурированном виде.

## 3. Анализ сценариев преднамеренных сбоев

Были отработаны три критических сценария отказа системы:

### Сценарий 1: Деградация базы данных (Database Corruption)
*   **Действие:** Запуск скрипта `break-db.sh`, удаляющего таблицы.
*   **Результат:** Сервис `app` начал возвращать `500 Internal Server Error`.
*   **SRE-наблюдение:** Несмотря на отказ основного приложения, `middleware` успешно продолжил фиксировать ошибки в файлы и базу (Dual Logging), что подтверждает независимость уровня наблюдаемости.

### Сценарий 3: Отказ зависимости (Service Dependency Failure)
*   **Действие:** Остановка контейнера `app` при работающем `rate-limiter`.
*   **Результат:** Nginx начал возвращать `502 Bad Gateway`, так как прокси-сервер не мог установить соединение с бэкендом.
*   **Диагностика:** Эндпоинт `/health` внутри самого `rate-limiter` остался доступен, что позволило локализовать проблему именно на уровне взаимодействия между сервисами, а не в самом прокси.

## Вывод
В результате выполнения была отработана методология поиска неисправностей на разных уровнях стека: от сетевых пакетов до логики взаимодействия микросервисов. Успешное восстановление системы после имитации сбоев (через `docker compose down -v` и рестарты) подтверждает готовность инфраструктуры к эксплуатации в реальных условиях.

## Ошибки:
1. Инциденты и деградация сервиса (Chaos Engineering): каскадный сбой (502 Bad Gateway) и нарушение целостности данных (500 Internal Server Error)
**Симптомы:** Веб-сервер Nginx перестал отдавать ответы от приложения, возвращая ошибку шлюза.
**Причина:** После ручных манипуляций с базой данных контейнер `rate-limiter` или основной сервис `app` перешли в состояние циклического перезапуска (CrashLoopBackOff) из-за невозможности установить корректное соединение с БД.
**Симптомы:** Появление ошибки `relation "users" does not exist` при обращении к API.
**Причина:** Использование скрипта `break-db.sh` в рамках сценария Chaos Engineering. Скрипт имитировал критическую аварию, удалив таблицу пользователей.
**Решение:** Выполнена полная очистка инфраструктуры и удаление томов командой `docker compose down -v`. Это позволило инициировать «холодный» старт базы данных и автоматическое выполнение скрипта `init-db.sql` для восстановления структуры таблиц.

2. Сетевые ошибки и ошибки SRE-инструментария: пустой захват трафика (tcpdump), синтаксическая ошибка CLI (Cycle Error)
**Проблема: Пустой захват трафика (tcpdump)**
**Симптомы:** Утилита `tcpdump` на порту DNS (53) не фиксировала пакетов при наличии запросов.
**Причина:** Отсутствие активных исходящих DNS-запросов из-за кэширования имен внутри Docker-сети либо использование протокола IPv6, который не попадал под узкий фильтр захвата.
**Симптомы:** Прерывание нагрузочного теста с ошибкой парсинга.
**Причина:** Некорректное использование точки с запятой и обратного слэша (`\;`) в цикле `for`. Оболочка воспринимала экранированный символ как текстовый аргумент для `curl`, а не как разделитель команд.
**Решение:** Исправление синтаксиса цикла на стандартный `; do ...; done`.
