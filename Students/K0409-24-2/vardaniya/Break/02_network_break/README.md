## Цель работы

Изучить восстановление сети в Linux после искусственной поломки маршрута, DNS и настроек разрешения имён.

## Ход работы

Сначала был запущен скрипт `02_network_break.sh`, который выполняет поломку сети:

- удаляет default route
- записывает неверный DNS в `/etc/resolv.conf`
- изменяет строку `hosts:` в `/etc/nsswitch.conf`

Команда запуска:

```bash
sudo bash 02_network_break.sh
```

Проверка, что сеть действительно сломана

После запуска скрипта были проверены:
	•маршруты
	•	содержимое /etc/resolv.conf
	•	строка hosts: в /etc/nsswitch.conf
	•	доступность сети через ping

Использованные команды:

ip route
cat /etc/resolv.conf
grep '^hosts:' /etc/nsswitch.conf
ping -c 2 8.8.8.8
ping -c 2 github.com


На скриншоте видно, что:
default route отсутствует
DNS подменён на неверный
строка hosts: изменена
сеть не работает

Восстановление сети

Для отката использовался скрипт восстановления:

sudo bash 99_restore_network.sh

После этого снова были проверены маршруты, DNS, nsswitch.conf и доступность сети.

ip route
cat /etc/resolv.conf
grep '^hosts:' /etc/nsswitch.conf
ping -c 2 8.8.8.8
ping -c 2 github.com

На скриншотах видно, что:
	•	default route восстановлен
	•	DNS снова рабочий
	•	строка hosts: восстановлена
	•	ping до 8.8.8.8 и github.com проходит успешно

Финальная проверка

После восстановления была выполнена проверка доступа к внешнему сайту:


curl -I --max-time 10 https://www.avito.ru


На скриншоте видно, что сервер отвечает HTTP/2 200.

Вывод

В ходе работы была искусственно нарушена конфигурация сети Linux, после чего была выполнена диагностика и восстановление:
	•	маршрутизации
	•	DNS
	•	правил разрешения имён

После восстановления сеть снова начала работать корректно.
